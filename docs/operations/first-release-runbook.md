# First Release Runbook

This runbook prepares and verifies the first controlled Quizz release, `v0.1.0`.

## Preconditions

Before creating `v0.1.0`, confirm:

- The [Repository Settings Checklist](repository-settings-checklist.md) is applied.
- `master` is protected.
- Required checks pass on `master`.
- CODEOWNERS review is enabled.
- GitHub Pages source is set to GitHub Actions.
- GHCR publishing uses `GITHUB_TOKEN`.
- No release tag named `v0.1.0` already exists.

Check for an existing remote tag:

```bash
git ls-remote --exit-code --tags origin v0.1.0
```

If that command finds `v0.1.0`, stop and inspect the existing release state before continuing.

## Local Verification Before Tagging

Start from the protected branch:

```bash
git checkout master
git pull --ff-only
```

Run local quality checks:

```bash
mvn spotless:check
mvn clean test
mvn -q javadoc:javadoc
python3 -m unittest discover scripts/demo/tests
mkdocs build --strict
```

Run Docker runtime checks:

```bash
docker compose config
docker compose build
docker compose up --build --wait
QUIZZ_HTTP_PORT="$(awk -F= '/^QUIZZ_HTTP_PORT=/{print $2; exit}' .env | tr -d '\r' | sed 's/^"//; s/"$//')"
curl -fsS "http://localhost:${QUIZZ_HTTP_PORT}/actuator/health"
docker compose down -v
```

Run local repository hooks:

```bash
pre-commit run --all-files
```

Run the commit message policy simulation if Node tooling is installed locally:

```bash
npx --yes @commitlint/cli@20.1.0 --config .commitlintrc.yaml <<< "chore: prepare v0.1.0 release"
```

Docker commands require Docker daemon access. The commitlint simulation requires local Node/npm/npx tooling.

## Create the Release Tag

Use annotated tags for release tags:

```bash
git tag -a v0.1.0 -m "v0.1.0"
git push origin v0.1.0
```

Do not move the tag after it is published.

## Observe the Release Workflow

After pushing the tag, watch `.github/workflows/release.yml` and confirm:

- Release validation starts from the tag.
- The tag reachability check passes against `origin/master`.
- Static audits pass.
- Maven, Python, Javadocs, MkDocs, and Docker health smoke pass.
- GHCR image publish succeeds.
- SBOM and provenance are generated by the Docker build.
- Artifact attestation succeeds.
- Published image pull verification succeeds.
- GitHub Release creation succeeds.

## Failure Handling

If validation fails before publishing:

1. Fix the issue on a side branch.
2. Open a pull request targeting `master`.
3. Wait for required checks and review.
4. Squash merge to `master`.
5. Create a replacement release tag only if no image or GitHub Release was published.

If the release was already published, do not move the tag. Cut a new patch release for the fix.

## Verify the Published Artifact

Pull the release image:

```bash
docker pull ghcr.io/0xmillennium/quizz:0.1.0
```

Inspect the image:

```bash
docker image inspect ghcr.io/0xmillennium/quizz:0.1.0
```

Verify the stable latest tag expectation:

```bash
docker pull ghcr.io/0xmillennium/quizz:latest
```

Verify the digest from the GitHub Release notes:

```bash
docker inspect --format='{{index .RepoDigests 0}}' ghcr.io/0xmillennium/quizz:0.1.0
```

Verify the attestation:

```bash
gh attestation verify oci://ghcr.io/0xmillennium/quizz:0.1.0 -R 0xmillennium/quizz
```

Verify the GitHub Release:

```bash
gh release view v0.1.0
```

Verify the GHCR package manually:

- Open the GitHub Packages page for `ghcr.io/0xmillennium/quizz`.
- Confirm the repository link.
- Confirm visibility.
- Confirm release tags are present.
- Confirm `latest` matches the expected stable release.

Verify GitHub Pages manually:

- Open the Pages URL shown in the Pages deployment output.
- Confirm the documentation home page loads.
- Confirm the release and delivery page exists.
- Confirm the repository settings checklist page exists.
